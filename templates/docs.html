System Architecture & Guides

Full technical blueprint of SheetBase. Discover how the platform ingests, processes, and stores complex electrical component specifications using artificial intelligence.

System Overview

SheetBase is a hybrid database management utility designed for electronic engineers. It resolves a crucial engineering bottleneck: manually extracting and parsing multi-page component datasheets into inventory lists.

By leveraging state-of-the-art Large Language Models (LLMs) paired with targeted parsing pipelines, SheetBase allows developers and engineers to upload raw PDF datasheets and automatically populate PostgreSQL tables with strict schema mapping.

Ingestion Pipeline

SheetBase implements a strict pipeline to translate unstructured PDF pages into safe SQL entries.

1

Temporary PDF Upload

Clients POST a raw PDF to `/upload-pdf`. The backend saves the document as a secure file named `temp_uploaded.pdf`.

2

Text Parsing & Page Limiting (PyMuPDF)

`fitz` opens the document and extracts the text contents. For safety under LLM token boundaries, we configure the extraction to limit to the **first 100 pages** (datasheet parameter tables and maximum ratings are always at the beginning, typically first 1-5 pages). 

pdf_text = extract_text_from_pdf(pdf_path, max_pages=100)
3

OpenRouter Prompt Assembly & LLM Routing

The text is injected into `prompt.txt`. The backend queries OpenRouter models sequentially, using a dynamic fallback mechanism to guarantee responses. 

# Dynamic model fallback chain
MODELS = [
    "openai/gpt-oss-120b:free",
    "meta-llama/llama-3.3-70b-instruct:free",
    "qwen/qwen3-next-80b-a3b-instruct:free"
]
4

SQL Sanitization & Transaction Ingestion

The returned SQL statement is stripped of markdown fences (` ```sql `). It is split by semicolons to execute separate statements sequentially inside an isolated database transaction, ensuring `rollback` if any error is encountered.

API Reference

SheetBase exposes three RESTful endpoints for frontend interactions and DB connectivity.

GET /data/<category>
Content-Type: application/json

Fetches all database entries for the requested category (`opv`, `bjt`, or `mosfet`).

POST /upload-pdf
Content-Type: multipart/form-data

Main entrypoint. Expects a PDF file under the name file. Initiates text extraction, runs the OpenRouter fallback, sanitizes statements, and writes parameters to the PostgreSQL database.

GET /test-db
Content-Type: application/json

Low-level connectivity test executing a minimal query (`SELECT 1`) to ensure PostgreSQL is online.

Database Schema

SheetBase maps component characteristics to three primary tables inside your PostgreSQL instance. 

CREATE TABLE opv (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    power_supply_voltage TEXT,
    input_offset_voltage TEXT,
    input_offset_current TEXT,
    input_common_mode_voltage_range TEXT,
    large_signal_open_loop_gain TEXT,
    input_bias_current TEXT,
    output_voltage_high_low_limit TEXT,
    output_source_current TEXT,
    power_supply_current TEXT
);
CREATE TABLE bjt (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    bjt_type TEXT CHECK (bjt_type IN ('NPN', 'PNP')),
    dc_current_gain_hfe TEXT,
    collector_emitter_voltage TEXT,
    collector_base_voltage TEXT,
    emitter_base_voltage TEXT,
    base_emitter_on_voltage TEXT
);
CREATE TABLE mosfet (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    mosfet_type TEXT CHECK (mosfet_type IN ('N-Kanal', 'P-Kanal')),
    drain_source_voltage TEXT,
    gate_source_voltage TEXT,
    continuous_drain_current TEXT,
    gate_threshold_voltage TEXT
);

Diagnostics & Connections

You can perform low-level diagnostic tests in the terminal to inspect database connectivity.

test-post.py

Inserts sample components (`OPA2134`, `2N2222A`, and `IRFZ44N`) with filled parameters into the database. Use this to verify insertion permissions. 

python3 test-post.py

test-conn.py

Tests the connection to the PostgreSQL instance at a raw socket level and pulls existing OPV contents. 

python3 test-conn.py

Security & Risk Audit

A transparency audit of the current architecture reveals critical risks and vulnerabilities to note in staging/production deployments.

Direct LLM SQL Execution (High Risk)

The pipeline parses and executes SQL returned by the LLM directly on your database. If a prompt or PDF is manipulated (e.g. prompt injection inside datasheet text), the model might generate malicious statements (like `DROP TABLE`).

Missing API Authentication & Rate Limits

Currently, `/upload-pdf` and database query endpoints are public. Anyone can execute operations, causing massive LLM expenses on your OpenRouter quota.

Recommended Upgrades

These modifications are highly recommended to upgrade SheetBase to a secure production-grade application.

SQL Validation Sandbox

Implement a SQL parsing step utilizing libraries like `sqlglot` to parse the LLM output and block statements containing `DROP`, `ALTER`, `DELETE`, or structural changes.

Refactor to ORM Inserts

Request the LLM to output a clean JSON payload instead of raw SQL. The backend can parse the JSON and use safe, parameterized SQLAlchemy ORM mappings to add items.

Background Task Queue (Celery)

Move document ingestion and OpenRouter queries into background workers. Return a task status ID to clients immediately, allowing the frontend to poll for completions without timing out.

API Protection Layer

Implement OAuth2 token validation, rate-limiting per API token, and strict CORS policies to protect endpoints from DDOS and quota exhaustion.
